# Group content by the user's job

Combine controls and explanations that serve one user job; split genuinely separate capabilities into separately legible features.

## Why

Grouping by implementation or chronology obscures what users can accomplish and why related controls belong together.

## Prefer

- Group controls, examples, and explanation around one coherent user job.
- Separate capabilities when they have different goals or outcomes.

## Avoid

- Grouping by implementation module when the user jobs differ.
- Splitting one job into several arbitrary feature fragments.

## Use when

- A page or document describes multiple controls or capabilities.

## Do not use when

- Side-by-side comparison of true peer capabilities is the job.

## Evidence

- Asked releases to split a combined capability into separately named features that match distinct user jobs.
- Asked dragging and docking to be one feature, while unrelated formula and tab improvements remain separate groups.

## Check

- Can each group be named as one user job?
- Do controls in the group share an outcome?
- Are distinct capabilities separately legible?
