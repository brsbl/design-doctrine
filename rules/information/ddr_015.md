# Make hierarchy reflect the job

Sibling surfaces with different jobs need visibly different hierarchy; sections should not merely look equally flat.

## Why

A consistent component system still needs enough hierarchy to communicate role and importance.

## Prefer

- Use hierarchy to distinguish primary work, supporting context, and metadata.
- Keep sibling styling consistent without erasing role differences.

## Avoid

- Giving every section the same weight because they share a component family.

## Use when

- Adjacent sections serve different user jobs or importance levels.

## Do not use when

- The sections are true peers that users compare directly.

## Evidence

- Asked visually sibling action sections to use the same hierarchy and questioned an unnecessary undocked footer boundary.
- Rejected flat Includes and Activity sections and asked for clearer organization.

## Check

- Can a skimmer tell which section owns the primary job?
- Are peers equal and non-peers differentiated?
- Does hierarchy remain within the shared system?
