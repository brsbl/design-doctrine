# Give related resources one shared spine

Related resource types should share a layout, taxonomy, and component spine, varying only where type, ownership, or state creates a real difference.

## Why

Shared structure improves learnability and prevents sibling resource systems from drifting into separate products.

## Prefer

- Reuse the same information skeleton and shared components.
- Express only genuine type-specific differences.

## Avoid

- Independent layouts, status vocabularies, and row structures for sibling resource types.

## Use when

- Several resource types live in one product hub or lifecycle.

## Do not use when

- A resource has a fundamentally different user job or lifecycle.

## Evidence

- Asked detail pages and resource rows to share taxonomy, layout, and components rather than invent parallel structures.
- Requested a reusable detail-page taxonomy based on what each resource's metadata means.

## Check

- Do sibling resource pages share the same structural grammar?
- Are differences explained by real resource semantics?
- Can one shared component own the common behavior?
