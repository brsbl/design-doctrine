# Write in user-capability language

Write for the user's task and product vocabulary rather than internal implementation; avoid overly technical or granular explanations.

## Why

Product language should help users act and understand value, not expose the structure of the implementation.

## Prefer

- Name the user job, capability, and observable outcome.
- Use product vocabulary consistently.

## Avoid

- Implementation terms that do not change the user's action.
- Granular process detail in primary copy.

## Use when

- Copy explains a feature, action, or result to the user.

## Do not use when

- The audience explicitly needs technical implementation detail.

## Evidence

- Asked release notes to avoid implementation identifiers and describe features in language meaningful to end users.
- Asked formula documentation to use ordinary terms and concise instructions instead of technical, granular wording.

## Check

- Does the copy lead with the user capability?
- Can internal terms be removed without losing meaning?
- Is the observable outcome clear?
