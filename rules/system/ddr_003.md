# Extend the existing product language

Start from the closest established product surface, component, behavior, and token set; extend it before introducing bespoke UI.

## Why

Product consistency is judged by perceived lineage and behavior, not merely by matching a few colors.

## Prefer

- Find the closest product-native sibling and reuse its structure, tokens, and behavior.
- Introduce a new primitive only when existing ones cannot express the job.

## Avoid

- Custom components whose anatomy or behavior feels unrelated to the host product.
- Copying surface color while inventing a different interaction language.

## Use when

- A new or changed surface has an established product neighbor or primitive.

## Do not use when

- Existing primitives cannot express the capability without creating a worse mental model.

## Evidence

- Required embed surfaces to reuse the editor's established preview and fullscreen behaviors.
- Flagged clipped comment hover treatment and a thread interface that looked like an unrelated mockup.
- Asked a drag affordance to preserve the row divider and reuse an existing fade treatment.
- Asked heavy action sections to match the closest established sibling pattern.

## Check

- What existing surface is the closest behavioral sibling?
- Which real tokens and components are reused?
- If a new primitive is necessary, is the reason explicit?
