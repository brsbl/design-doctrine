# Demonstrate work with current realistic evidence

Design evidence must use the current product UI and realistic inputs, framed to demonstrate the user job; reject fake, stale, low-resolution, or cramped captures.

## Why

Evidence is part of the design: an unrealistic or stale demonstration prevents an honest judgment of whether the work succeeds.

## Prefer

- Use current product UI with realistic safe content.
- Frame the state and viewport around the user job.
- Inspect final-resolution assets.

## Avoid

- Fake or placeholder product UI.
- Stale, low-resolution, cropped, or cramped captures that hide the behavior.

## Use when

- A visual artifact is used to approve, document, or claim a design outcome.

## Do not use when

- A rough conceptual sketch is explicitly the requested stage and is labeled conceptual.

## Evidence

- Requested real, matching examples across feature visuals and removal of decorative or floating UI that distorted the demonstration.
- Requested current, high-resolution, uncrowded screenshots that show the relevant surrounding interface.
- Rejected prototype-only, broad visual changes and required minimal changes be applied to the real component stories.

## Check

- Does the artifact come from the current product state?
- Are inputs realistic and safe to publish?
- Does framing demonstrate the intended user job?
- Was final resolution inspected?
