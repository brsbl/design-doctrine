# Keep durable decisions editable at the source

For durable decisions in a note, default to editable Markdown tables or checklists; use HTML for a focused visual or interaction demonstration rather than as an opaque replacement.

## Why

A visually rich but non-editable decision surface can displace the user's actual work instead of supporting it.

## Prefer

- Keep the decision itself in editable, commentable source.
- Use scoped HTML only where visual shape or interaction materially improves understanding.

## Avoid

- Replacing editable choices with a non-editable visual artifact.

## Use when

- The artifact captures decisions the user will revise or comment on.

## Do not use when

- The requested deliverable is primarily a visual or interactive demonstration.

## Evidence

- Rejected using chat as a substitute for editing the underlying note.

## Check

- Can the user edit the actual decision at its canonical source?
- Is any HTML block scoped to one visual job?
